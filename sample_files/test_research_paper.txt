Scalable Async Document Processing with FastAPI and Celery
==========================================================

Abstract
--------
This paper presents a production-grade architecture for high-throughput document
ingestion pipelines leveraging message queues, background workers, and real-time
progress streaming via Server-Sent Events. Benchmarks show sub-4-second median
processing latency for typical business documents under concurrent load.

Keywords: async processing, celery, redis, document pipeline, fastapi, sse

1. Introduction
---------------
Modern enterprise platforms demand asynchronous workflows for document-heavy
operations. Blocking request threads on CPU- or I/O-bound document tasks
degrades throughput and user experience. This paper describes DocFlow, a
reference implementation that combines FastAPI, Celery, Redis Pub/Sub, and
PostgreSQL to deliver decoupled, observable, and horizontally scalable document
processing.

2. Architecture
---------------
The system is organized into four distinct layers:
  - API Layer:         FastAPI handles HTTP ingestion, authentication, and SSE
  - Queue Layer:       Redis serves as both the Celery broker and Pub/Sub channel
  - Worker Layer:      Celery executes document tasks in isolated background jobs
  - Persistence Layer: PostgreSQL stores documents, job state, and extracted results

3. Performance Evaluation
-------------------------
Load tests were conducted with 250 concurrent upload clients using k6.
Median processing time was 3.8 s; p99 was 7.2 s for documents up to 10 MB.
Worker auto-scaling (via Celery autoscale) kept queue depth below 20 at peak.

4. Conclusion
-------------
The proposed architecture sustains 250+ concurrent uploads with predictable
latency. Future work includes integrating an LLM-based extraction layer for
richer semantic metadata and adding multi-tenant job isolation.
